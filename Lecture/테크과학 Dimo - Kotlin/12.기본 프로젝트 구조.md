# 프로젝트 구조

### **프로젝트(Project)**

코틀린으로 애플리케이션을 만들 때, 관련된 모든 내용을 담는 가장 큰 단위이다. 하나의 프로젝트 안에 여러 개의 모듈(Module)을 포함할 수 있다.

### **모듈(Module)**

프로젝트를 이루는 하위 단위이다. 직접 작성할 수도 있고, 이미 만들어진 라이브러리 모듈을 가져와 사용할 수도 있다.

모듈 안에는 다음과 같은 것들이 포함될 수 있다

- 코틀린 소스 코드 파일
- 리소스 파일 (이미지, 설정 파일 등)
- 모듈 관련 설정 파일

즉, 모듈은 하나의 독립 실행 단위 또는 재사용 가능한 기능 단위로 볼 수 있다.

# 논리적 구조 : 패키지

실제 물리적인 구조(폴더/파일)와 달리, 소스 코드의 소속을 지정하기 위한 논리적 단위이다. 패키지를 사용하면 이름 충돌을 방지할 수 있고, 코드를 기능별로 체계적으로 관리할 수 있다.

### 네이밍 규칙

일반적으로 **도메인 이름을 거꾸로 배치**한 후, 프로젝트명과 기능명을 붙여서 사용한다.

- 예: youtube.com → com.youtube
- 프로젝트 이름과 기능별 하위 패키지를 붙여서 확장한다.
    
    ```kotlin
    package com.youtube.dimo
    ```
    
- 패키지를 명시하지 않으면 자동으로 default 패키지에 속한다.

### 패키지 선언

코드 파일의 맨 위에 package를 명시한다.

```kotlin
package com.youtube.dimo
```

- 패키지명을 기준으로 코드가 소속된다.
- 코틀린은 자바와 달리 패키지명과 폴더 구조가 반드시 일치할 필요가 없다.
- 단지 파일 상단에 선언된 패키지 이름을 기준으로 컴파일러가 관리한다.

### import 구문

다른 패키지에 속한 클래스나 함수를 사용하려면 import가 필요하다.

```kotlin
package com.youtube.dimo

import com.youtube.dimo.base  // 다른 패키지의 요소 사용
```

- import를 통해 외부 패키지에 있는 변수, 함수, 클래스 등을 그대로 사용할 수 있다.
- 만약 이름이 충돌한다면, 풀 네임을 사용해야 한다.

# 자바와 코틀린

- 코틀린은 자바와 달리 클래스명과 파일명이 일치하지 않아도 된다.
- 하나의 파일에 여러 개의 클래스를 넣어도 문제없이 컴파일된다.
- 파일/폴더 구조가 아닌, package 키워드를 기준으로 코드가 구분되기 때문이다.
